# Krok 10d — WorldCalendarConfig: Design Spec (Fáze 4.1)

**Datum vzniku:** 2026-05-06 (extrahováno z původního sdruženého specu 10cd, viz [Předchůdce](#předchůdce))
**Stav:** Připraveno pro Fázi 4.1 (čeká na ukončení Fáze 3.x)

---

## Předchůdce

Tento spec byl původně součástí `2026-05-04-krok-10cd-timeline-calendar-config-design.md`, kde se kombinoval s Timeline. Při Fázi 3.2 (timeline implementace) jsme spec rozdělili — timeline modul jde samostatně bez calendar configu, calendar přijde v 4.1 a retroaktivně začne plnit `celestialStates` v Timeline GET response.

**Důsledek pro Timeline:** Žádná schema změna při 4.1. Stačí přidat `WorldCalendarConfigService` do `TimelineModule.imports` a změnit `celestialStates: []` placeholder na reálný výpočet. Frontend je forward-compat (`celestialStates` field už existuje).

---

## Přehled

Modul `world-calendar-config` pro fantasy kalendář světa — rok rozdělený na měsíce, dny v týdnu, nebeská tělesa s periodami (měsíc, slunce, planety, komety), referenční datum.

**Path API:** `/api/worlds/:worldId/calendar-config`
**Modul (kód):** `backend/src/modules/world-calendar-config/`

**Vztah k jiným modulům:**
- 1:1 per svět (jeden config per `worldId`)
- Exportuje `WorldCalendarConfigService` pro `TimelineModule` (computed `celestialStates`)
- Možná budoucí konzumenti: GameEvents (calendar-aware scheduling), WorldWeather (sezónní logika)

## WorldCalendarConfig

### Schema

Kolekce `world_calendar_configs`, 1:1 per world.

```typescript
interface WorldCalendarConfig {
  id: string
  worldId: string                  // unique index
  hoursPerDay: number              // výchozí 24
  daysOfWeek: string[]             // názvy dní v týdnu (může být prázdné)

  months: {
    name: string                   // např. "Měsíc plamenů"
    daysCount: number
  }[]

  celestialBodies: CelestialBody[]

  referenceDate: {
    year: number
    month: number                  // 1-based index do months[]
    day: number
    hour: number                   // 0-based
  } | null

  createdAt: Date
  updatedAt: Date
}
```

### CelestialBody

```typescript
type CelestialBodyType = 'moon' | 'sun' | 'planet' | 'comet' | 'other'

interface CelestialBody {
  id: string                       // crypto.randomUUID() — klíč pro overrides na TimelineEvent
  name: string
  type: CelestialBodyType
  config: MoonConfig | SunConfig | PlanetConfig | CometConfig | OtherConfig
  referenceState: string           // stav tělesa na referenceDate
}

interface MoonConfig {
  cycleLength: number              // délka cyklu v dnech
  phases: string[]                 // pojmenované fáze, např. ['nový', 'dorůstající', 'úplněk', 'couvající']
}

interface SunConfig {
  riseHour: number[]               // hodina východu per měsíc (délka = months.length)
  setHour: number[]                // hodina západu per měsíc (délka = months.length)
}

interface PlanetConfig {
  orbitalPeriod: number            // oběžná doba v dnech
  constellations: string[]         // konstelace/znamení (planeta se pohybuje skrz ně)
}

interface CometConfig {
  periodYears: number              // perioda v rocích světa
  apparitionDurationYears: number  // jak dlouho je viditelná při průletu
}

interface OtherConfig {
  cycleLength: number
  states: string[]                 // pojmenované stavy cyklu
}
```

### Výpočet absolutního dne

```
totalDaysPerYear = sum(months[].daysCount)

absoluteDay(year, month, day) =
  year * totalDaysPerYear
  + sum(months[0 .. month-2].daysCount)
  + day

referenceAbsoluteDay = absoluteDay(referenceDate.year, referenceDate.month, referenceDate.day)
```

### Výpočet stavu tělesa

`referenceState` je string odpovídající jedné z pojmenovaných hodnot tělesa (fáze, konstelace, stav). Při výpočtu se převede na `referenceOffset` (číslo dní od začátku cyklu):

```
// Moon / Other:
referenceOffset = phases.indexOf(referenceState) * (cycleLength / phases.length)

// Planet:
referenceOffset = constellations.indexOf(referenceState) * (orbitalPeriod / constellations.length)

// Comet: referenceState = 'viditelná' | 'neviditelná'
// pokud 'viditelná': referenceOffset = 0 (začátek průletu)
// pokud 'neviditelná': referenceOffset = apparitionDays (první den mimo průlet)

// Sun: nemá cycleLength, referenceState se ignoruje
```

Stav na libovolném datu:

```
delta = absoluteDay(year, month, day) - referenceAbsoluteDay

// Moon / Other:
phaseIndex = ((delta + referenceOffset) % cycleLength + cycleLength) % cycleLength
state = phases[floor(phaseIndex / (cycleLength / phases.length))]

// Planet:
positionDeg = ((delta + referenceOffset) % orbitalPeriod / orbitalPeriod * 360 + 360) % 360
state = constellations[floor(positionDeg / (360 / constellations.length))]

// Comet:
totalPeriodDays = periodYears * totalDaysPerYear
apparitionDays = apparitionDurationYears * totalDaysPerYear
phaseInCycle = ((delta + referenceOffset) % totalPeriodDays + totalPeriodDays) % totalPeriodDays
state = phaseInCycle < apparitionDays ? 'viditelná' : 'neviditelná'

// Sun:
state = `vychod: ${riseHour[month-1]}:00, zapad: ${setHour[month-1]}:00`
```

### Záporné delta (datum před referenceDate)

JS `%` na záporném čísle vrací záporek — proto `((x % n) + n) % n` aby zaručilo non-negative výsledek. Toto musí být **explicitně testováno** (TDD test: 14 dní před referencí měsíce s `referenceState='nový'` → po 28-denním cyklu očekáváme `úplněk`, tj. delta=-14, půl cyklu před nový).

## API

```
GET  /api/worlds/:worldId/calendar-config
PUT  /api/worlds/:worldId/calendar-config       # upsert (full replace)
```

### Auth

| Endpoint | Auth |
|---|---|
| `GET` | JWT + member světa (`WorldRole >= Hrac`) **\|** Admin/Superadmin shortcut |
| `PUT` | JWT + Admin/Superadmin **\|** `WorldRole >= PomocnyPJ` na `worldId` |

**Konzistence s WorldNews + Timeline (3.2):** stejný pattern `assertCanWrite`/`assertMember` z `world-news.service.ts` a `timeline.service.ts`. Pomocný PJ může editovat (per memory rule + Fáze 3.1 rozhodnutí).

### Request body PUT

```ts
// UpsertWorldCalendarConfigDto
{
  hoursPerDay?: number             // 1..48 default 24
  daysOfWeek?: string[]            // prázdné OK
  months?: {                       // alespoň 1 měsíc pro funkční calendar
    name: string                   // 1..50
    daysCount: number              // >= 1
  }[]
  celestialBodies?: {
    id?: string                    // pokud chybí, server přiřadí UUID
    name: string                   // 1..50
    type: 'moon' | 'sun' | 'planet' | 'comet' | 'other'
    config: object                 // typed dle `type` (validace v service)
    referenceState: string         // pro sun ignorováno
  }[]
  referenceDate?: {
    year: number
    month: number                  // 1..months.length
    day: number                    // 1..months[month-1].daysCount
    hour: number                   // 0..hoursPerDay-1
  } | null
}
```

### Validace

| Pravidlo | Vrstva | Chyba |
|---|---|---|
| `hoursPerDay` 1..48 | DTO | 400 |
| `months[].name` non-empty, ≤ 50 | DTO | 400 |
| `months[].daysCount` ≥ 1 | DTO | 400 |
| `celestialBodies[].type` ∈ enum | DTO | 400 |
| `SunConfig.riseHour.length === months.length` | Service | 400 |
| `SunConfig.setHour.length === months.length` | Service | 400 |
| `referenceDate.month` ∈ `1..months.length` | Service | 400 |
| `referenceDate.day` ∈ `1..months[month-1].daysCount` | Service | 400 |
| `referenceDate.hour` ∈ `0..hoursPerDay-1` | Service | 400 |
| `worldId` neexistující (PUT) | Service | 403 (anti-leak) / 404 (GET) |

### Response

GET vrací plný `WorldCalendarConfig` nebo `null` pokud config ještě nebyl nastaven (nový svět).

### Integrace s Timeline (po 4.1)

V `TimelineModule.imports` přidat `WorldCalendarConfigModule`. V `TimelineService` injectnout `WorldCalendarConfigService`. V `findMany`/`findById`/`create`/`update` upravit `toResponse(...)`:

```ts
// PŘED (Fáze 3.2): celestialStates: []
// PO (Fáze 4.1):
const config = await this.calendarConfigService.getConfig(event.worldId);
const celestialStates = config
  ? this.calendarConfigService.calculateCelestialStates(
      event.year, event.month, event.day, config, event.celestialOverrides,
    )
  : [];
```

**Žádná schema změna v Timeline. Žádný frontend breaking change.**

## Testy

### `world-calendar-config.utils.spec.ts` — pure logic

- `totalDaysPerYear` — součet dní všech měsíců
- `absoluteDay` — rok 0/měsíc 1/den 1 = 1; rok 0/měsíc 2/den 1 = 31 (pro 30-denní leden); rok 1/měsíc 1/den 1 = 91 (pro 3 měsíce po 30 dnech)
- `calculateCelestialStates`:
  - prázdné `celestialBodies` → `[]`
  - chybí `referenceDate` → `[]`
  - moon: stejné datum jako reference vrátí `referenceState`
  - moon: 7 dní po novém s 28-denním cyklem 4 fází → dorůstající
  - **moon záporné delta**: 14 dní před `nový` → `úplněk`
  - manuální override (`isManualOverride: true`)
  - sun: hodiny rise/set per měsíc
  - comet: viditelná v průletovém okně → neviditelná po
  - planet: konstelace dle orbital period

### `world-calendar-config.service.spec.ts`

- `getConfig` vrátí null pro nenastavený svět
- `upsertConfig` jako Admin/Superadmin → OK
- `upsertConfig` jako PomocnyPJ světa → OK (konzistence s WorldNews/Timeline)
- `upsertConfig` jako Korektor → 403
- SunConfig délka neodpovídá `months.length` → 400
- `celestialBodies` bez `id` → server přiřadí UUID
- Neexistující svět (PUT) → 403 (anti-leak)
- `referenceDate.month` mimo rozsah → 400
- `referenceDate.day` mimo rozsah měsíce → 400

### `world-calendar-config.controller.ts`

- 401 bez JWT
- 403 jako non-member (GET)
- 200 jako Admin (GET)

## Zpětný retrofit Timeline (po 4.1)

Po implementaci 4.1 vytvořit follow-up task:

1. Přidat `WorldCalendarConfigModule` do `TimelineModule.imports`
2. Injectnout `WorldCalendarConfigService` do `TimelineService`
3. Upravit `toResponse()` helper — místo prázdného `celestialStates: []` volat `calendarConfigService.calculateCelestialStates(...)`
4. Aktualizovat existující `timeline.service.spec.ts` — mockovat `WorldCalendarConfigService`, ověřit že `celestialStates` jsou plněny pro events ve světě s configem a zůstává `[]` pro svět bez configu
5. Doplnit test: `celestialOverrides` přebije výpočet (`isManualOverride: true`)
6. Roadmap: zaškrtnout 4.1 + retrofit checkbox

## Mimo scope

- **Frontend** kalendář picker / overview UI
- **Validace** že `referenceState` odpovídá pojmenovaným hodnotám tělesa (server akceptuje libovolný string; chybný se projeví jako index `-1` → fallback na 0)
- **Migrace** z JSON exportů — calendar config je nový (parita se starým systémem ano, ale data v staré DB neexistují per audit)
- **Versioning** changes (zachovat historii configu) — pokud nový spec si vyžádá, řešit zvlášť
