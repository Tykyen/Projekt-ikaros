# Fáze 2.4 — Drobné endpointy z Kroku 16b — Design

> Spec pro 4 endpointy odhalené v auditu jako chybějící. Brainstorming 2026-05-06, schváleno před psaním plánu.

## Kontext

Audit `roadmap2.md` § 2.4 vypsal 4 endpointy chybějící v backendu. Jsou to drobné dotahovky — žádné nové schéma, jen API povrch nad existujícími daty.

| Endpoint | Modul | Účel |
|---|---|---|
| `PUT /api/worlds/:worldId/calendarconfig` | worlds | Zápis `WorldCalendarConfig` (PJ+/Admin) |
| `POST /api/admin/users` | admin | Vytvoření uživatele adminem |
| `GET /api/users/exists/:username` | users | Anon endpoint pro registrační formulář |
| `PUT /api/users/:id/theme` | users | Dedikovaný theme endpoint (merge) |

## Pravidla autorizace (důležité)

**Vlastník světa NENÍ automaticky autorizován.** Autorizace na world-level mutace se odvozuje výhradně z `WorldRole ≥ PomocnýPJ` ve `WorldMembership` + globální role `Admin`/`Superadmin`. Toto pravidlo platí pro `calendarconfig` endpoint v této specifikaci.

Existující `worlds.service.ts:245` a `:260` má opačnou logiku (dovoluje vlastníka). Tato spec to **neopravuje** (mimo scope) — ale nový endpoint se vyhne stejné chybě.

## 1. `PUT /api/worlds/:worldId/calendarconfig`

### Kontrakt
- **Path:** `worldId` — Mongo ObjectId existujícího světa.
- **Body (`UpdateCalendarConfigDto`):** full `WorldCalendarConfig` objekt. Pole zachová strukturu z `world.interface.ts:39` (`calendarConfig?: WorldCalendarConfig`). Replace semantika — celé pole se přepíše body-em.
- **Auth:** `JwtAuthGuard` + autorizační kontrola v service:
  - `requester.role ≤ Admin` (globální Admin/Superadmin) → ✅
  - jinak: `WorldMembership` pro `(worldId, requester.id)` musí mít `role ≥ WorldRole.PomocnýPJ` → ✅
  - jinak → 403
- **Response:** 200 — updated `World` entity.
- **Errors:**
  - 400 — neplatný DTO (class-validator)
  - 401 — chybí JWT
  - 403 — neoprávněn (viz výše)
  - 404 — svět neexistuje

### Implementace
- Controller: `worlds.controller.ts` — nový endpoint:
  ```ts
  @Put(':worldId/calendarconfig')
  @UseGuards(JwtAuthGuard)
  updateCalendarConfig(
    @Param('worldId') worldId: string,
    @Body() dto: UpdateCalendarConfigDto,
    @CurrentUser() user: RequestUser,
  )
  ```
- Service: `WorldsService.updateCalendarConfig(worldId, config, requester)`:
  - načti `world` z repo, 404 pokud null
  - autorizace: viz výše (`requester.role ≤ Admin` OR `membership.role ≥ PomocnýPJ`)
  - delegace na `worldsRepo.updateCalendarConfig(worldId, config)`
  - vrátí updated world
- Repository: `IWorldsRepository.updateCalendarConfig(worldId, config)` — Mongo `$set: { calendarConfig: config }`.
- DTO: `update-calendar-config.dto.ts` v `worlds/dto/` — `class-validator` schéma podle `WorldCalendarConfig` interface.

## 2. `POST /api/admin/users`

### Kontrakt
- **Path:** žádný param.
- **Body (`CreateUserAdminDto`):**
  - `username: string` — required, unique (case-sensitive — viz § 3 pro důvod)
  - `email: string` — required, unique, IsEmail (lowercase per schéma)
  - `password: string` — required, plain (hash dělá service), min length per existující `RegisterDto`
  - `role?: UserRole` — default `Hrac` (per schéma default)
- **Auth:** `JwtAuthGuard` + `AdminGuard` (= `requester.role ≤ Admin`).
- **Response:** 201 — `SafeUser` (bez `passwordHash`).
- **Errors:**
  - 400 — neplatný DTO
  - 401 — chybí JWT
  - 403 — neadmin
  - 409 — username nebo email konflikt

### Implementace
- Controller: `admin.controller.ts` — nový endpoint:
  ```ts
  @Post('users')
  @UseGuards(AdminGuard)
  createUser(@Body() dto: CreateUserAdminDto)
  ```
- Service: `AdminService.createUser(dto)`:
  - `usersRepo.findByUsername(dto.username)` → conflict → throw `ConflictException`
  - `usersRepo.findByEmail(dto.email)` → conflict → throw `ConflictException`
  - hash password (bcrypt jako v `auth.service`)
  - `usersRepo.create({ ...dto, passwordHash, role: dto.role ?? UserRole.Hrac })`
  - return `stripPassword(user)`
- DTO: `create-user-admin.dto.ts` v `admin/dto/` — class-validator (IsString, IsEmail, IsEnum, MinLength).

## 3. `GET /api/users/exists/:username`

### Kontrakt
- **Path:** `username` — string, max length 64.
- **Auth:** ŽÁDNÁ — anon endpoint.
- **Lookup:** case-sensitive (= `findByUsername` aktuální chování). Konzistence s registrací: `username` schéma nemá `lowercase: true`, takže `Karel` a `karel` jsou různí users; `exists` musí toto chování zrcadlit, jinak by frontend dostával false-positive "volné" a register by selhal.
- **Response:** 200 — `{ exists: boolean }`.
- **Errors:**
  - 400 — username delší než 64 znaků (preventivní guard, ne hard schema limit)

### Implementace
- Controller: `users.controller.ts` — nový endpoint **PŘED** `@Get(':id')` (jinak by se `exists` chytilo jako `:id`):
  ```ts
  @Get('exists/:username')
  exists(@Param('username') username: string)
  ```
- Service: `UsersService.exists(username)`:
  - validace: `username.length <= 64` → jinak `BadRequestException`
  - `repo.findByUsername(username)` → vrátí `{ exists: !!user }`

### Známé limitace (mimo scope)
- ⚠️ `username` v `user.schema.ts:12` nemá `lowercase: true` (na rozdíl od `email` na ř. 9). Potenciální zdroj UX bugů (uživatelé zapomenou casing). Doporučení: revize a migrace v jiné fázi.
- Žádný throttling. Endpoint je vystavený k user-enumeration. Login má stejný problém — řešení patří do globální fáze rate-limitingu, ne do 2.4.

## 4. `PUT /api/users/:id/theme`

### Kontrakt
- **Path:** `id` — userId.
- **Body (`UpdateThemeDto`):** `{ themeSettings: Record<string, unknown> }`.
- **Merge semantika:** service načte usera, sloučí `existing.themeSettings` s `dto.themeSettings` (object spread, nové klíče přepíšou stávající), uloží. Klíče v existujícím objektu, které nejsou v body, zůstávají (např. `calendarMonth` zachová předchozí hodnotu).
- **Auth:** `JwtAuthGuard` + buď `requester.id === id`, nebo `requester.role ≤ Admin`. Jinak 403. Stejný pattern jako existující `updateCalendarMonth`.
- **Response:** 200 — updated `SafeUser`.
- **Errors:**
  - 400 — neplatný DTO
  - 401 — chybí JWT
  - 403 — neoprávněn
  - 404 — uživatel neexistuje

### Implementace
- Controller: `users.controller.ts` — nový endpoint **PŘED** `@Patch(':id')` (vyhnout se kolizi):
  ```ts
  @Put(':id/theme')
  @UseGuards(JwtAuthGuard)
  updateTheme(
    @Param('id') id: string,
    @Body() dto: UpdateThemeDto,
    @CurrentUser() requester: Requester,
  )
  ```
- Service: `UsersService.updateTheme(id, themeSettings)`:
  - load user (404 pokud null)
  - merged = `{ ...user.themeSettings, ...themeSettings }`
  - `repo.update(id, { themeSettings: merged })`
  - return safe user
- DTO: `update-theme.dto.ts` v `users/dto/` — `@IsObject() themeSettings`.

### Vztah k `updateCalendarMonth`
Stávající `PUT /users/updateCalendarMonth/:id` zůstává **beze změny**. `theme` endpoint je obecnější souběžně. Frontend si vybere — pro úpravu jen calendarMonth použije specializovaný endpoint, pro plný theme patch použije `theme`.

## Test plán (spec testy)

### worlds.service.spec.ts (rozšíření)
- `updateCalendarConfig`:
  - PomocnýPJ ve světě → ✅ replace
  - PJ ve světě → ✅ replace
  - Hráč ve světě → 403
  - Globální Admin (bez membershipu) → ✅
  - Vlastník bez membershipu → 403 (NEW behavior — záměrné odlišení od existujícího kódu)
  - Neexistující svět → 404

### admin.service.spec.ts (rozšíření)
- `createUser`:
  - Happy path: hash password, default role Hrac, vrátí safe user
  - Username konflikt → 409
  - Email konflikt → 409
  - Explicit role v DTO → respect

### admin.controller.spec.ts (NEW nebo rozšíření existujícího)
- AdminGuard:
  - Anon → 401
  - Hráč → 403
  - Admin → ✅

### users.service.spec.ts (rozšíření)
- `exists`:
  - Existující username → `{ exists: true }`
  - Neexistující → `{ exists: false }`
  - Case-sensitive: `Karel` existuje, `karel` neexistuje → `{ exists: false }` pro `karel`
  - Username > 64 znaků → BadRequestException
- `updateTheme`:
  - Merge zachová `calendarMonth` při patch jiného klíče
  - Replace klíče: nový hodnotou přepíše starý
  - Neexistující user → 404

### users.controller.spec.ts (NEW nebo rozšíření)
- `updateTheme` auth:
  - Sám uživatel → ✅
  - Jiný uživatel → 403
  - Admin → ✅
- `exists` anon: žádný guard, vrací 200 i bez JWT
